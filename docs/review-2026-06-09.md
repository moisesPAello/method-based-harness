## Issue A ‚Äî `doctor` baseline writer clobbers the `init` verify audit (bug)

`_audit_verify_gate` (cli.py:236-243) merges into an existing `.harness/baseline.json`
(`existing["verify"] = audit`), but `_write_baseline` (cli.py:517-518) ‚Äî run by `doctor`
by default ‚Äî rebuilds the dict from scratch, silently dropping the `verify` key. Running
`init` then `doctor` loses the verify-gate audit (terminated/red/green) the reviewer is
supposed to diff against. The two writers also disagree on schema (`verify` vs `gates`).

Also: `_write_baseline` calls `subprocess.run` directly (cli.py:513) while `_run_verify`
exists precisely to be injectable ‚Äî any doctor-with-baseline test today would actually
spawn the profile's `sync_check`.

**Fix:** unify both writers on one merge-into-existing baseline writer with one schema;
make the gate runner injectable; add a doctor-after-init test asserting `verify` survives.
(Tests currently miss this because every doctor test passes `no_baseline=True`.)

## Issue B ‚Äî `init` and `upgrade` disagree on the methodology/host source of truth (bug)

- `cmd_init` trusts `--methodology/--host` flags (cli.py:294,307) and ignores the
  profile's `methodology:`/`host:` keys; `cmd_upgrade` reads them FROM the profile
  (cli.py:351-352). On mismatch, init installs X and the first upgrade silently
  re-renders as Y.
- `--from-profile` clobbers an existing `.harness/profile.yaml` without `--force`: the
  clash guard (cli.py:317-323) only covers `result.files`; cli.py:327 writes the profile
  unconditionally.
- Dry-run output doesn't match real writes: init's `planned` (cli.py:308) omits the local
  scaffold files and `baseline.json`; upgrade's dry-run never mentions the
  `.claude/CLAUDE.md` block merge.

**Fix:** when a profile exists, read methodology/host from it (error/warn on flag
mismatch); extend the clash guard to the profile target; make dry-run enumerate
everything the real run writes.

## Issue C ‚Äî interpreter/verify handling is inconsistent across audit, render, doctor

- The verify audit pins the interpreter only for commands starting with bare
  `python`/`python3` (cli.py:212-213) ‚Äî but the canonical case is `pytest -q` with
  `interpreter: .venv/bin/python` (the bundled example). The audit runs PATH `pytest`
  while `leader.md` says "run `pytest -q` (via `.venv/bin/python`)" ‚Äî audit can be green
  where the agents' run is red.
- Fallback styles diverge: `profile.get("interpreter", "python3")` (claude.py:73,220,
  cli.py:566) vs `profile.get("interpreter") or "python3"` (cli.py:208). A profile with
  `interpreter: ""`/null renders `Bash( *)` permissions and "via ``" in leader.md while
  the audit quietly uses python3.
- Doctor resolvability checks are asymmetric: the interpreter check also tries
  repo-relative paths (cli.py:568), the verify check only `shutil.which` (cli.py:570-571)
  ‚Äî `verify.command: .venv/bin/pytest` warns spuriously.

**Fix:** one shared `interpreter(profile)` helper; pin pytest-style commands consistently
(or document why not); symmetric resolvability checks.

## Issue D ‚Äî `merge_block` corrupts on half-deleted markers; doctor never checks the block

compile.py:87-94: if a user deletes just one marker, the block is appended again
(duplicate BEGIN); if END precedes BEGIN, the slicing scrambles content. `doctor` checks
agent frontmatter but never the CLAUDE.md block's marker integrity.

**Fix:** make `merge_block` defensive (one-marker and inverted-marker cases), add a
doctor check for marker presence/sanity, with tests.

## Issue E ‚Äî CLI UX: dead flags and raw tracebacks on hand-edited files

- `-v/--verbose` and `-q/--quiet` (cli.py:657-658) are defined, shown in `--help`, and
  never read by any handler.
- `cmd_status` does unguarded `json.loads` (cli.py:459); `_load_yaml` lets
  `yaml.YAMLError` escape from init/upgrade/doctor. These are exactly the files users
  hand-edit most (`feature_list.json`, `profile.yaml`). `_load_features` already shows
  the fail-soft pattern.

**Fix:** either implement or remove -v/-q; wrap profile/feature-list parsing in a clean
one-line error + EX_FAIL.

## Issue F ‚Äî settings.json render: clobber-on-adopt and inconsistent permission patterns

- `.claude/settings.json` is a plain managed write: adopting with `--force` clobbers a
  user's existing settings (CLAUDE.md, by contrast, gets a marked-block merge), and the
  uninstall doc says to delete the whole file.
- Permission patterns (claude.py:221-225): `Bash({interp} *)` (with space) vs
  `Bash({verify_tok}*)` (without) ‚Äî `Bash(pytest*)` also matches `pytest-anything`, and
  `Bash(python3 *)` won't match bare `python3`.

**Fix:** at minimum warn when the pre-existing settings.json is non-harness before
overwrite (or JSON-merge); normalize the permission pattern shapes.
